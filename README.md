# Fitness CRM Tests

Автотесты для проверки CRM системы тренера.

---

## Что покрыто:

### UI тесты (Selenium)

* логин (успешный / неуспешный)
* создание клиента
* открытие клиента
* logout
* проверка пустых полей логина

### API тесты

* авторизация
* работа с клиентами

---

## Стек:

* Java
* Selenium
* JUnit 5
* Gradle
* k6 (нагрузочное тестирование)

---

## Конфигурация окружения

В проекте устранена жёсткая привязка (hardcode) к URL.

Ранее базовый адрес приложения был зашит напрямую в тестах.
Теперь используется централизованная конфигурация:

* создан класс `TestConfig`
* базовый URL вынесен в `BASE_URL`
* тесты используют конфиг

Это позволяет запускать тесты на разных окружениях (dev / stage / prod) без изменения кода.

## Нагрузочное тестирование (k6)

Нагрузочные тесты находятся в папке:

```text
load/
```

---

### 1. Smoke-тест (главная страница)

Файл:

```text
load/smoke-home.js
```

Параметры:

* 3 виртуальных пользователя (VUs)
* длительность: 10 секунд

Результат:

* 0% ошибок
* среднее время ответа ~180–200 ms

Вывод:

Главная страница работает стабильно и быстро под лёгкой нагрузкой.

---

### 2. API тест (регистрация + получение клиентов)

Файл:

```text
load/api-register-clients.js
```

Сценарий:

1. регистрация нового пользователя
2. получение session cookie
3. авторизованный запрос `/api/clients`

Параметры:

* 5 виртуальных пользователей
* 5 итераций

Результат:

* 0% ошибок
* стабильная работа API
* среднее время ответа ~2 секунды

Вывод:

API работает корректно, но время ответа достаточно высокое.

## Ограничения тестирования

Нагрузочное тестирование проводилось на бесплатном тарифе Vercel (Hobby).

Ограничения:

* лимиты по CPU и памяти
* ограничения на количество одновременных запросов
* возможные задержки (cold start)
* shared инфраструктура

Поэтому тестирование выполнялось с ограниченной нагрузкой (до 5–10 пользователей).

Для полноценного нагрузочного тестирования требуется:

* отдельное окружение (stage)
* платный тариф или выделенные ресурсы

---

## Особенность авторизации

При тестировании под нагрузкой с одним пользователем возникали ошибки:

```text
401 Unauthorized
```

Причина:

* backend поддерживает только одну активную сессию на пользователя
* при новом логине предыдущая сессия инвалидируется

Решение:

* использовать уникальных пользователей в нагрузочных тестах
